# 0.1.9A — UI Forge Sync

**Release date:** 2026-04-28
**Previous version:** 0.1.9
**Version bump:** none — sub-release tracked as 0.1.9A
**Target:** UI Forge ≥ 0.1.9

## Summary

Brings StackShift up to date with UI Forge sub-releases 0.1.9A → 0.1.9D shipped after StackShift 0.1.9. Adds three protocols, surfaces UI Forge's new `--handoff` and PostToolUse hook capabilities, centralizes the cross-skill handshake into one canonical document, and reduces duplication in skill-root resolution.

All changes are additive. Existing 0.1.9 projects continue to work unchanged. No version bump — this is tracked as the 0.1.9A sub-release.

---

## New Protocols

### Paired-Mode Contract (recommended tier)

**File:** `skills/stackshift-core/protocols/paired-mode-contract.md`

Single canonical reference for the StackShift ↔ UI Forge handshake. Centralizes content previously scattered across multiple files: skill-root resolution order, marker fields (`.stackshift/installed.json` vs `design/design-arch.json`), the optional `_paired` mirror block, the flag refusal matrix (`--creative` / `--diff` / `--preview`), modifier composition rules, and the contract version handoff.

Other paired protocols (`accessibility`, `brand`, `claude-design-handoff`, `auto-verify-hook`, `variant-router`) link to this doc instead of restating its rules. The protocol introduces no new behavior — it documents existing invariants in one place so changes are reviewable atomically.

### Claude Design Handoff (optional tier)

**File:** `skills/stackshift-core/protocols/claude-design-handoff.md`

Activates UI Forge's `+CLAUDE_DESIGN` modifier and the `--handoff <url>` flag introduced in UI Forge 0.1.9D. When installed, Step 4d's ref-selection rules permit a Claude Design URL as a layout source — mutually exclusive with HTML/TSX layout refs.

Bootstrap (Step 6g, conditional) offers to run `/forge-export-design` to seed Claude Design with project tokens before any handoff is created. Bootstrap also adds `design/.handoff-cache/` and `design/claude-design-bundle/` to the default `.forgeignore` (Step 8).

**Step 4 postcondition (when active):** Generated FORGE NOTES header must contain a `CLAUDE_DESIGN` sub-block.

Companion reference: `skills/stackshift-core/references/claude-design-roundtrip.md` — full six-step round-trip workflow.

### Auto-Verify Hook (optional tier)

**File:** `skills/stackshift-core/protocols/auto-verify-hook.md`

Wires UI Forge's `verify.js` single-arg mode (introduced in UI Forge 0.1.9B) as a Claude Code `PostToolUse` hook. When installed, every `.tsx` write triggers automatic contract validation; UI Forge's single-arg mode silently no-ops on non-FORGE files, so the hook is safe project-wide.

Bootstrap (Step 7b, conditional) idempotently merges the hook entry into `.claude/settings.json`. The hook resolves the contract path via the `// @contract <path>` directive that UI Forge writes on line 3 of FORGE NOTES.

This protocol is Claude Code-specific — bootstrap skips Step 7b on Codex CLI / general agents and informs the user the hook is a no-op outside Claude Code.

## Workflow Updates

### `${UI_FORGE_SKILL_DIR}` resolution delegates to UI Forge `detect.sh`

**Files:** `skills/stackshift-core/workflow/4-variants.md`, `skills/stackshift-core/bootstrap/install.md`, `skills/stackshift-core/protocols/paired-mode-contract.md`

UI Forge 0.1.9 ships `scripts/detect.sh` as the upstream source of truth for the cross-agent skill-root search order. Both Step 4 and bootstrap Step 6c now invoke `detect.sh` first when available, falling back to the existing 7-entry path lookup if the script is missing (UI Forge < 0.1.9).

The fallback list is duplicated across three files (`workflow/4-variants.md`, `bootstrap/install.md`, `protocols/paired-mode-contract.md`) — each carries a comment that the others must stay in sync.

### Slash-command-first invocation in Claude Code

**File:** `skills/stackshift-core/workflow/4-variants.md`

Step 4d's invocation block now documents two paths:

- Claude Code (preferred when `.claude/` exists): `/forge --task ... --refs ... --signal CONVERT_VARIANT --mode body-only`
- Codex CLI / general agents / UI Forge < 0.1.9: bare `node ${UI_FORGE_SKILL_DIR}/scripts/invoke.js ...` (unchanged)

A third invocation block documents the Claude Design handoff variant: `/forge --handoff <url> ...`. `--task` may be omitted — UI Forge derives it from the handoff README heading.

### Postconditions extended

**File:** `skills/stackshift-core/workflow/4-variants.md`

Step 4d's postcondition list now includes:

- `// @contract <path>` directive on line 3 of FORGE NOTES (manual check; required by UI Forge ≥ 0.1.9 `SIGNAL_VARIANT`).
- `BRAND` sub-block when the `brand` protocol is active (was previously implicit; now explicit for symmetry with `A11Y` and `CLAUDE_DESIGN`).
- `CLAUDE_DESIGN` sub-block when the `claude-design-handoff` protocol is active and `--handoff` was used.
- No handoff-generated CSS classes inline in TSX (every token resolves to `design-arch.json`).

When the `auto-verify-hook` protocol is active, the structural `validate-contract.js` postcondition collapses into "hook exit code 0" — the manual command remains valid as a backup.

### Failure modes extended

**File:** `skills/stackshift-core/workflow/4-variants.md`

Three new rows:

- Missing `// @contract` directive — checker StackShift; max 1 retry.
- Claude Design handoff fetch failed — checker UI Forge `fetch-handoff.js` stderr; halt; no auto-retry.
- Missing `CLAUDE_DESIGN` sub-block — checker StackShift; max 1 retry.

### Step 3 — `// @contract` directive note

**File:** `skills/stackshift-core/workflow/3-types.md`

Added a paragraph documenting that `@contract-version` (JSDoc tag on the props interface) and `// @contract <path>` (line 3 of FORGE NOTES) work as a pair — the JSDoc tag tracks version drift; the directive lets UI Forge's `verify.js` resolve the contract path without a manual argument.

### Checklist — protocol-conditional rows

**File:** `skills/stackshift-core/workflow/checklist.md`

New "Protocol-conditional" section. Each block fires only when its protocol is in the materialized set:

- `accessibility` active → `A11Y` sub-block on every variant; `a11yRequired: true` in marker.
- `brand` active → `BRAND` sub-block; `designStandards.brand` resolves; `brand.md` non-starter.
- `claude-design-handoff` active and used → `CLAUDE_DESIGN` sub-block; cache/bundle in `.forgeignore`; no inline handoff classes.
- `auto-verify-hook` active → hook entry in `.claude/settings.json`; hook fired without errors.

Step 4 base checklist also gains:

- FORGE NOTES contains `// @contract <path>` directive on line 3.
- `design/design-arch.json` no older than 7 days (UI Forge warns at 8+).

## Bootstrap Updates

### Step 6c — `--schema-v4` opt-in and stderr capture

**File:** `skills/stackshift-core/bootstrap/install.md`

When the user opts to run `scan.js` and the `accessibility` protocol (or any custom dark-mode protocol) is selected, bootstrap prompts:

```
? Extract dark-mode Tailwind tokens during scan? (y/N)
```

If yes, `--schema-v4` is appended. The choice is recorded under `uiForgeIntegration.schemaV4` in `.stackshift/installed.json`.

Bootstrap also captures stderr from `scan.js`. UI Forge 0.1.9C emits a `═`-bordered banner if AI synthesis falls back to static analysis — bootstrap surfaces the captured banner in the Step 9 summary so users know `design-arch.json` `patterns.*` may be coarse.

### Step 6g — Claude Design bundle export (conditional)

**File:** `skills/stackshift-core/bootstrap/install.md`

Runs only when the `claude-design-handoff` protocol is in the materialized set and `design/design-arch.json` exists. Prompts the user; on confirmation runs `/forge-export-design` (Claude Code) or `node ${UI_FORGE_SKILL_DIR}/scripts/export-design.js` (other runtimes). Result recorded under `uiForgeIntegration.bundleExported`.

### Step 6h — `_paired` mirror block

**File:** `skills/stackshift-core/bootstrap/install.md`

Once `design/design-arch.json` exists, bootstrap writes a `_paired` block mirroring StackShift markers UI Forge cares about:

```json
{
  "_paired": {
    "stackshiftVersion": "0.1.9",
    "a11yRequired": true,
    "contractVersion": "1.0.0",
    "protocols": ["variant-router", "accessibility"]
  }
}
```

`.stackshift/installed.json` remains the canonical write target; the `_paired` block is read-optimization only. Older UI Forge versions ignore unknown top-level keys, so it's forward-compatible.

### Step 7b — PostToolUse hook (conditional)

**File:** `skills/stackshift-core/bootstrap/install.md`

Runs only when the `auto-verify-hook` protocol is in the materialized set and Claude Code runtime is detected (`.claude/` directory or `$CLAUDE_PLUGIN_ROOT`). Idempotently merges a `PostToolUse` matcher with `Write|Edit` and command `node ${UI_FORGE_SKILL_DIR}/scripts/verify.js "$CLAUDE_FILE_PATH"` into `.claude/settings.json`. Preserves any unrelated user-written hook entries.

Result recorded under `uiForgeIntegration.postToolUseHook` in `.stackshift/installed.json`.

### Step 8 — `.forgeignore` defaults extended

**File:** `skills/stackshift-core/bootstrap/install.md`

Default `.forgeignore` now includes `design/.handoff-cache/` and `design/claude-design-bundle/` so UI Forge's `scan.js` never walks regeneratable Claude Design artifacts. Existing projects with a hand-authored `.forgeignore` are not modified — the file is never overwritten after first write.

### Step 9 — Summary expanded

**File:** `skills/stackshift-core/bootstrap/install.md`

The summary now includes lines for `design/claude-design-bundle/`, `.claude/settings.json` (hook), `schemaV4 (dark tokens)`, and `_paired mirror block`. If UI Forge's scan emitted a fallback banner, the captured banner is appended verbatim under a `Scan synthesis fallback:` heading.

### Canonical `installed.json` shape extended

**File:** `skills/stackshift-core/bootstrap/install.md`

`uiForgeIntegration` gains `schemaV4`, `bundleExported`, and `postToolUseHook` fields. Field ownership table updated.
